# 要件定義

## 1. プロジェクト概要

海外のIT・AI系メディアを横断的に収集し、AIが「編集者」として

- 読む価値のある記事の選定(今日読むべき5本)
- 自然な日本語タイトルへの翻訳
- 数十秒で理解できる日本語要約
- タグ付けによる整理

を行うニュースリーダー。ユーザーは英語を読めなくても、毎日数分で海外の重要ニュースをキャッチアップできる。

### 提供価値(最重要)

> 海外IT・AIニュースの重要ポイントを、日本語で短時間に理解できること

全文翻訳サービスではない。「読む負担」を減らし「知る価値」を最大化する。

## 2. ターゲットユーザー

| ペルソナ | 課題 | 本サービスでの解決 |
|---|---|---|
| 日本のソフトウェアエンジニア | 英語記事を読むのが負担。巡回先が多い | 1画面で重要ニュースを日本語で把握 |
| AI/技術動向を追うPM・企画職 | 何が重要ニュースか判断できない | AIによる「今日読むべき5本」 |
| 学生・技術学習者 | 一次情報にアクセスする習慣がない | 元記事リンクで一次情報への入口を提供 |

## 3. ユースケース

- **UC-1: 朝のキャッチアップ** — ユーザーはトップ画面を開き、「今日読むべき5本」を読む。所要時間 2〜3分。気になる記事は元記事リンクから原文へ。
- **UC-2: 最新記事の流し読み** — 5本の下に続く最新記事一覧(日本語タイトル+要約)をスクロールして流し読みする。
- **UC-3: 一次情報への到達** — 要約で概要を掴んだ後、`[元記事を読む]` から原文記事に遷移して詳細を確認する。

## 4. 機能要件

### 4.1 記事収集(バッチ)

| ID | 要件 | 優先度 |
|---|---|---|
| F-01 | 登録済みメディアのRSS/Atomフィードを定期取得する | Must |
| F-02 | 記事の重複登録を防止する(URL/GUIDベース) | Must |
| F-03 | 対象メディアは設定(DB)で追加・無効化できる | Must |
| F-04 | フィードに本文がない場合、記事ページから本文相当のテキストを取得する(要約精度のため) | Should |
| F-05 | 取得失敗したメディアはスキップし、他メディアの処理を継続する | Must |

### 4.2 AI処理(バッチ)

| ID | 要件 | 優先度 |
|---|---|---|
| F-10 | 英語タイトルを意味の伝わる自然な日本語タイトルに翻訳する | Must |
| F-11 | 記事を「何が起きたか/なぜ重要か/誰に関係するか」の観点で日本語要約する(数十秒で読める分量) | Must |
| F-12 | 記事に定義済みタグセットからタグを付与する(0〜5個) | Must |
| F-13 | 1日1回、当日の記事から「今日読むべき5本」を選定し、選定理由を付す | Must |
| F-14 | AI出力はスキーマ検証(JSON指示+zod検証。APIモード時は Structured Outputs 併用)し、不正な出力は再試行する | Must |
| F-15 | AI処理失敗記事は原文タイトルのまま表示可能とする(サービス全体を止めない) | Should |

### 4.3 表示(Web)

| ID | 要件 | 優先度 |
|---|---|---|
| F-20 | トップ画面に「今日読むべき5本」を表示する | Must |
| F-21 | その下に最新記事一覧を新しい順で表示する | Must |
| F-22 | 各記事カードに以下を表示する: メディア名 / 日本語タイトル / AI要約 / 公開日時 / 元記事リンク | Must |
| F-23 | 記事一覧はページネーションまたは無限スクロールで過去分を閲覧できる | Should |
| F-24 | 元記事リンクは新規タブで開き、出典が明確に分かる表示とする | Must |
| F-25 | モバイル・デスクトップ両対応(レスポンシブ) | Must |
| F-26 | タグを記事カードに表示する(タグ検索はMVP対象外) | Could |

### 4.4 MVPでやらないこと(明示的な非要件)

ユーザー認証 / AIチャット / コメント / SNS機能 / いいね / フォロー / パーソナライズ推薦 / 記事全文の翻訳表示。全ユーザー共通のニュース体験を提供する。詳細は [03_mvp-scope.md](./03_mvp-scope.md)。

## 5. 非機能要件

| 分類 | 要件 |
|---|---|
| 性能 | トップ画面の表示は 1秒以内(AI処理はすべて事前バッチ。リクエスト時にLLMを呼ばない) |
| 更新頻度 | 記事収集・AI処理: 1日3回(朝・昼・夕。設定で増減可)。「今日の5本」: 1日1回(JST 朝 7:00 目安) |
| 可用性 | 個人プロジェクト水準。バッチ失敗時も前回データで閲覧継続可能なこと |
| コスト | **AI追加費用 $0/月**(Claude Pro サブスク枠で実行)+ インフラ無料枠のみ。AI呼び出しはアダプタで抽象化し、Anthropic API へ移行可能な設計を維持 |
| 拡張性 | メディア追加はDBレコード追加のみで対応。将来機能(既読・タグ検索等)を阻害しないスキーマ |
| 運用性 | バッチ実行結果(成功/失敗件数・コスト)をログで追跡できる |
| 法令順守 | 要約は元記事の代替とならない分量に留め、出典・リンクを必ず明示([09参照](./09_legal-operational-concerns.md)) |

## 6. 用語定義

| 用語 | 定義 |
|---|---|
| メディア(source) | 記事の収集元(OpenAI Blog, TechCrunch 等) |
| 記事(article) | フィードから取得した1エントリ |
| ダイジェスト処理 | タイトル翻訳+要約+タグ付けの一括AI処理 |
| 今日の5本(daily picks) | AIが1日1回選定するおすすめ記事(ランキングではなく「編集長のおすすめ」) |
