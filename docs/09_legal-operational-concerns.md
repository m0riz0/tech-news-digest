# 法的・運用上の懸念事項

> 本ドキュメントは設計判断のためのリスク整理であり、法的助言ではない。公開・収益化の前に必要に応じて専門家の確認を推奨。日本法(著作権法)を前提に整理する。

## 1. 著作権リスク

### 1.1 要約の「翻案」該当リスク — 最重要

- 記事の**要約は原著作物の翻案(著作権法27条)に該当しうる**。特に原文の表現・構成をなぞった詳細な要約(いわゆる抄録)はリスクが高い
- 一方、**事実(何が起きたか)それ自体は著作物ではない**。「事実の抽出+自分の言葉での短い紹介」に近づけるほどリスクは下がる

**設計上の対策(実装に反映済み)**:

| 対策 | 反映箇所 |
|---|---|
| 要約は2〜4文の短文に制限し、元記事の代替とならない分量に留める | digester 出力スキーマ([04 §3.3](./04_architecture.md)) |
| 「何が起きたか/なぜ重要か/誰に関係するか」という**事実+独自の観点**での再構成をプロンプトで指示(原文の逐語的な縮約を避ける) | digester プロンプト |
| 全文翻訳は行わない(BRIEFの非要件でもある) | 要件 F-群 |
| 元記事リンク・メディア名・公開日時を必ず表示し、**元記事への送客が前提**のUIにする | F-22, F-24 |
| 取得した本文テキストは表示せず、AI入力の内部利用のみ。処理後のクリア運用も選択肢 | [05 §2.2](./05_database-design.md) |

補足: AI学習・情報解析目的の複製には権利制限規定(30条の4)があるが、**生成した要約を公衆に提供する行為はその範囲外**。「短い・事実ベース・送客前提」という要約自体の性質で守る必要がある。

### 1.2 タイトル翻訳

- ニュース見出しは一般に創作性が低く著作物性が否定されやすい(読売オンライン見出し事件など)が、**無断利用が不法行為と評価された例はある**
- 本サービスは見出しの転載ではなく「内容を伝える日本語見出しの新規作成」であり、直訳の転載よりリスクは低い。プロンプトでも「直訳ではなく意味が伝わるタイトルを生成」と指示する(これは品質要件とも一致)

### 1.3 画像・サムネイル

- **MVPでは記事画像を表示しない**。OGP画像等の転載は複製権侵害リスクが明確に高い。将来も favicon 程度に留めるか、権利処理を確認してから

## 2. 利用規約・アクセスマナー

| 項目 | 方針 |
|---|---|
| RSS/Atom | フィードの公開は購読目的の利用を予定した提供と解されるが、各メディアの利用規約を導入時に個別確認する(sources 追加時のチェックリスト化) |
| 本文取得(補助) | robots.txt を尊重。`User-Agent` にサービス名と連絡先を明示。取得は要約生成の内部利用に限定し、リクエスト間隔を空ける(同一ホストへ数秒間隔) |
| クローラー拒否メディア | AIクローラーをブロックする意思表示(robots.txt, ToS)をしているメディアは本文取得をせず、フィード内テキストのみで要約(品質が足りなければ対象から外す) |
| Anthropic規約 | Usage Policy に従う(ニュース要約は通常利用の範囲)。**MVP期は Pro サブスクリプション枠(Claude Code ヘッドレス)で実行するため、消費者向け利用規約の「個人利用」の範囲で運用する** — 自分専用の運用テストという現方針はこの範囲内と整理。収益化・公開規模拡大の際は Anthropic API(商用利用前提の規約)へ移行する([04 §8参照](./04_architecture.md#8-ai実行アダプタとapi移行パス)) |

## 3. 削除依頼・権利者対応

- **About ページに運営者連絡先と削除依頼窓口を設置**(MVP必須。[03 §3](./03_mvp-scope.md))
- 権利者から申し立てがあった場合のフロー: 該当記事の非表示(`sources.is_active` off または記事単位の非表示フラグ)→ 誠実に協議。争わず即時対応を原則とする
- 記事単位の非表示に備え、articles に `is_hidden` フラグを実装時に追加検討(または削除で対応)

## 4. AI出力の品質リスク

| リスク | 内容 | 対策 |
|---|---|---|
| 誤訳・誤要約 | 原文と異なる内容を伝えてしまう(報道内容の歪曲) | ①要約に断定的な追加解釈を加えない指示 ②「AIによる要約であり正確性は元記事を確認」の免責表示(全カード共通のフッター/Aboutに明記) |
| ハルシネーション | 本文にない事実の混入 | プロンプトで「本文に書かれていることのみ」と制約。Structured Outputs で形式は保証されるが内容は定期的に抜き取り確認 |
| 選定の偏り | 特定メディア・特定トピックへの偏重 | curator プロンプトにメディア分散の制約。picks の履歴をSQLで確認できる |
| 不適切コンテンツ | 元記事自体が不正確・扇動的 | 対象メディアを信頼できるものに限定(ホワイトリスト方式が本設計の前提) |

## 5. 運用リスク

| リスク | 内容 | 対策 |
|---|---|---|
| サブスク枠の圧迫 | バッチが Pro プランのレート枠(5時間ウィンドウ+週次上限)を消費し、普段の Claude Code 利用と競合する | ①約10記事を1プロンプトに集約(1日7プロンプト程度に抑制) ②フィード大量流入時(初回取得・障害復旧)に備え1バッチの処理記事数に上限(例: 100件)、初回投入は `published_at` 直近N日以内のみ ③`batch_runs` で実行記録・レート制限エラーを監視 |
| サブスク前提の変化 | Pro プランのレート制限・ヘッドレス利用条件・料金の変更 | AI実行はアダプタ抽象化済み。条件が悪化したら `AI_PROVIDER=api` へ切替(その際のコスト試算は [04 §7.2](./04_architecture.md#7-コスト見積もり)) |
| フィード仕様変更・廃止 | メディア側のRSS終了・URL変更 | `fetch_logs` で連続失敗を検知。メディア単位で無効化できる設計 |
| モデルの廃止・移行 | サブスク既定モデルの変更 / APIモデルのdeprecation | 出力はzod検証で品質下限を担保。APIモード時のモデルIDは環境変数化済み。移行時は数記事でA/B確認 |
| バッチ基盤の制約 | GitHub Actions の無料枠(2,000分/月)・スケジュール遅延 | 1日3回×数分で枠に余裕。多少の実行遅延は日次ダイジェストの性質上許容 |
| DB肥大 | content_text の蓄積 | 処理完了後のクリア運用 / 保持期間ポリシー(例: 90日) |
| 認証情報漏洩 | CLAUDE_CODE_OAUTH_TOKEN / ANTHROPIC_API_KEY 等 | GitHub Secrets 管理・クライアント非露出・リポジトリに含めない(.env.example のみコミット)。漏洩時は再発行 |

## 6. プライバシー・その他

- ユーザー認証なし・ユーザーデータ収集なしのため、個人情報関連の義務は最小(アクセス解析を入れる場合はプライバシーポリシーに明記)
- サービス名・ロゴは既存商標と衝突しないか公開前に確認
- 広告・収益化を行う場合は要約利用の商用性が権利リスク評価に影響するため、**収益化前に本ドキュメントを再レビュー**すること。あわせて収益化時は ①AI実行を Anthropic API モードへ移行(サブスクの個人利用範囲を超えるため) ②Vercel を Pro プラン化(Hobby は非商用条件)の2点がセットで必要
