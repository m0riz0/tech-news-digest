# システムアーキテクチャ

## 1. 設計方針

1. **リクエスト時にAIを呼ばない** — すべてのAI処理は事前バッチで完了させ、Webはただの「DBを読むサイト」にする。表示は速く、AI利用量は記事数にのみ比例する。
2. **AI実行はアダプタで抽象化する** — MVP期は契約中の Claude Pro プラン枠(Claude Code ヘッドレスモード)で実行し、**AI追加費用 $0/月** を実現する。成長時(公開拡大・収益化)は環境変数の切替だけで Anthropic API モードへ移行できる([§8参照](#8-ai実行アダプタとapi移行パス))。
3. **パイプラインは段階ごとに独立** — 収集 / AI処理 / 選定 を分離し、どこかが失敗しても他は動く。
4. **状態はすべてDBに持つ** — バッチはステートレス。記事の処理状態(pending → processed / failed)をDBで管理し、再実行はいつでも安全(冪等)。
5. **小さく始める** — MVPはモノリシックな Next.js アプリ+cron 実行のバッチスクリプトで十分。キューやワーカー基盤は導入しない。

## 2. 全体構成図

```
                        ┌─────────────────────────────────────────┐
                        │              バッチ処理層                │
   ┌──────────┐  ①収集  │  ┌─────────┐  ②AI処理  ┌─────────────┐  │
   │ 各メディア │───────▶│  │ fetcher │─────────▶│  digester    │  │
   │ RSS/Atom │         │  └─────────┘           │ (Claude API) │  │
   └──────────┘         │       │                └─────────────┘  │
                        │       │      ③日次選定  ┌─────────────┐  │
                        │       │      ─────────▶│  curator     │  │
                        │       │                │ (Claude API) │  │
                        │       ▼                └──────┬──────┘  │
                        └───────┼───────────────────────┼─────────┘
                                ▼                       ▼
                        ┌─────────────────────────────────────────┐
                        │            PostgreSQL (DB)               │
                        │  sources / articles / tags / daily_picks │
                        └────────────────────┬────────────────────┘
                                             │ read only
                        ┌────────────────────▼────────────────────┐
                        │       Next.js (SSR/ISR + API)            │
                        │  トップ画面(今日の5本 + 最新記事一覧)   │
                        └────────────────────┬────────────────────┘
                                             │
                                        ユーザー(Web)
```

## 3. バッチパイプライン

### 3.1 ジョブ一覧

| ジョブ | 頻度 | 処理内容 |
|---|---|---|
| `fetch` (fetcher) | 1日3回(JST 朝7:00・昼・夕方。設定で増減可) | 全アクティブメディアのフィード取得 → 新規記事を `pending` で保存 |
| `digest` (digester) | fetch 直後に連続実行 | `pending` 記事をAI処理(翻訳+要約+タグ) → `processed` に更新 |
| `curate` (curator) | 1日1回(朝7:00の回のみ) | 過去24時間の `processed` 記事から「今日の5本」を選定 |

MVPでは 1 つの cron エントリポイント(`fetch → digest`、朝の回のみ `curate` も)として実装してよい。実行基盤は GitHub Actions のスケジュール実行([07参照](./07_tech-stack.md))。

### 3.2 fetcher(収集)

```
for source in active_sources:
    feed = fetch(source.feed_url)          # 失敗したら記録してcontinue
    for entry in feed.entries:
        if exists(entry.guid or url): skip # 冪等性: GUID/URLで重複排除
        content = entry.content or entry.summary
        if content が短すぎる:
            content = fetch_article_text(entry.url)  # 補助: 本文抽出(Should要件)
        insert article(status='pending', content_text=truncate(content))
```

- 本文取得は robots.txt / 利用規約を尊重し、要約生成のための内部利用に限定(本文自体は表示しない)。詳細は [09](./09_legal-operational-concerns.md)。
- `content_text` はAI入力用に上限(例: 先頭 8,000 文字)で切り詰めて保存。

### 3.3 digester(AIダイジェスト処理)

**約10記事を1プロンプトにまとめて**、翻訳・要約・タグ付けを一括で行う(Pro プランのレート枠を節約。1日あたり digester 5〜6プロンプト+curator 1プロンプト程度で収まる想定)。

**入力**: 記事ごとの { id, 英語タイトル, 本文テキスト, メディア名 } の配列(最大10件)
**出力(JSON配列。zod でスキーマ検証)**:

```json
[
  {
    "article_id": 123,
    "title_ja": "日本語タイトル(直訳ではなく意味が伝わる見出し)",
    "summary_ja": "2〜4文の要約。何が起きたか/なぜ重要か/誰に関係するか",
    "tags": ["LLM", "OpenAI"],
    "importance": 4,
    "importance_reason": "重要度判断の根拠(1文)"
  }
]
```

設計ポイント:

- **出力の構造化**: プロンプトでJSON出力を指示し、**zod でスキーマ検証**する。検証失敗・欠落記事があれば該当分のみ1回再試行(APIモード時は Structured Outputs でスキーマ準拠を強制できるため、zod は二重チェックになる)
- **タグは定義済みリストからの選択**とし、タグの無限増殖を防ぐ。リストはDB管理(`tags` テーブル)でプロンプトに注入。zod 検証で未知タグを弾く
- **importance(1〜5)を記事単位で事前スコアリング**しておき、curator の一次絞り込みに使う
- 失敗時は指数バックオフで最大2回再試行 → それでも失敗なら `failed` として原文タイトルで表示可能にする(F-15)
- 1プロンプトあたりの本文入力は記事ごとに上限(先頭 8,000 文字)で切り詰め、コンテキストを抑える

**実行モードは2つ**(`LLMClient` アダプタで共通化。[§8参照](#8-ai実行アダプタとapi移行パス)):

| モード | 方式 | 位置付け |
|---|---|---|
| A: Claude Code ヘッドレス | `claude -p`(サブスクOAuth認証)を子プロセス実行し、JSON出力を受け取る | **既定(MVP)**。AI追加費用 $0。Pro のレート枠(5時間ウィンドウ+週次上限)を消費 |
| B: Anthropic API | `@anthropic-ai/sdk` の Messages API(必要に応じ Batch API で50%オフ) | 成長時(公開拡大・収益化)。従量課金だがレート枠・規約の制約から解放 |

### 3.4 curator(今日の5本)

1日1回、**1プロンプト**で選定する。

**入力**: 直近72時間に公開された記事の { id, メディア名, 日本語タイトル, 要約, importance, 公開日時 } のリスト
(候補が多い場合は importance 上位で 30〜50 件に事前絞り込み)

候補ウィンドウの設計(当日限定にしない・かつ固定化させない):

- **ウィンドウは72時間** — 当日取得分に限定せず、直近の影響度の高い記事を対象にする(バッチが1日止まってもトップが痩せない)
- **過去に選定済みの記事は候補から除外** — daily_picks の履歴とNOT EXISTSで機械的に除外し、高importance記事が5本の枠を占領し続けるのを防ぐ。大きな話題は続報記事として再登場できる
- 結果として各記事は「最大3日間候補になり、選ばれたら卒業、選ばれないまま72時間経過で落選」という動きになる
- 当日分の picks は除外しない(同日のバッチ再実行で洗い替え可能にするため)
- プロンプトで「重要度が同程度なら公開日時が新しい記事を優先」を指示し、ウィンドウ内でも鮮度を優先する

**出力(JSON。digester と同様に zod で検証)**:

```json
{
  "picks": [
    { "article_id": 123, "rank": 1, "reason": "選定理由(読者向けの1文)" }
  ]
}
```

設計ポイント:

- プロンプトで「ランキングではなく編集長のおすすめ」という役割と選定基準(業界への影響 / 技術的重要度 / 新規性 / エンジニアへの関連性 / 話題性)を明示
- 同一メディア偏重を避ける制約をプロンプトに含める
- 記事が5本未満の日はある分だけ選定
- 選定結果は `daily_picks` テーブルに日付付きで保存(アーカイブが将来機能でそのまま使える)

### 3.5 AIモデル選定

| モード | モデル | 備考 |
|---|---|---|
| A: Claude Code(既定) | サブスクの既定モデル(Sonnet 系) | 追加費用ゼロ。モデル指定はせずサブスクの標準に従う(要約・翻訳品質は運用テストで確認) |
| B: Anthropic API(成長時) | 環境変数で指定(`DIGEST_MODEL` / `CURATE_MODEL`) | 品質重視なら `claude-opus-4-8`、コスト重視なら `claude-sonnet-5` / `claude-haiku-4-5`。移行時に数記事でA/B確認 |

### 3.6 プロンプトキャッシュ戦略(APIモード時のみ)

prompt caching / Batch API は Anthropic API 固有の最適化のため、**モードB移行時に適用**する:

- digester のシステムプロンプト(役割定義+タグリスト+出力ルール)は固定化し、`cache_control: {type: "ephemeral"}` を付与
- 記事本文・タイトルなど可変部は必ずキャッシュポイントより後(user メッセージ)に置く
- モードAでもプロンプトを「固定部+可変部」に分けて実装しておけば、移行時にそのままキャッシュ境界になる

## 4. Web層

- **Next.js(App Router)** のサーバーコンポーネントでDBから直接読み出し
- トップ画面は **ISR(再検証間隔 5〜10分)** または CDN キャッシュ。バッチ更新頻度(1日3回)より十分短ければよく、リクエストごとのDBアクセスも不要になる
- API Routes は将来のクライアント機能(無限スクロール等)向けに最小限提供([06参照](./06_api-design.md))

## 5. エラーハンドリング / 冪等性

| 障害 | 挙動 |
|---|---|
| フィード取得失敗 | 該当メディアをスキップ、`fetch_logs` に記録。連続失敗はログで検知 |
| AI呼び出し失敗(レート制限/エラー) | 指数バックオフで再試行。Pro枠の5時間ウィンドウ上限に当たった場合はそのバッチを打ち切り、次回実行で `pending` 分から再開(冪等なので安全)。最終失敗は `failed` ステータス |
| AI出力の不正(スキーマ違反) | zod 検証で検出し、該当記事のみ1回再試行(APIモード時は Structured Outputs で構造的に防止) |
| バッチ多重起動 | 記事ステータス(`pending` → `processing` → `processed`)の更新を排他的に行い二重処理を防止 |
| curator 失敗 | 当日の picks は前日分を表示し続ける(トップが空にならない) |

## 6. セキュリティ

- Web層は読み取り専用(公開データのみ)。認証なしで攻撃面が小さい
- バッチのトリガーエンドポイントを設ける場合はシークレットトークンで保護(GitHub Actions のスクリプト直接実行なら不要)
- `CLAUDE_CODE_OAUTH_TOKEN`(サブスク認証)/ `ANTHROPIC_API_KEY`(APIモード時)/ DB接続情報は GitHub Secrets 等で管理。クライアントに露出しない

## 7. コスト見積もり

### 7.1 MVP(モードA: Claude Code ヘッドレス)— 既定

**AI追加費用: $0/月**(契約済み Pro プラン内)。金銭コストの代わりに Pro のレート枠(5時間ウィンドウ+週次上限)を消費する。

前提: 10メディア × 平均5記事/日 = 約50記事/日、10記事/プロンプト。

| 項目 | 消費量 |
|---|---|
| digester | 5〜6プロンプト/日(朝・昼・夕の3回に分散) |
| curator | 1プロンプト/日 |
| **合計** | **約7プロンプト/日** — Pro枠(目安: 数十プロンプト/5時間)に対して十分小さい |

- **注意**: この枠は自分が普段使う Claude Code と共有。バッチは1日3回・各数プロンプトに集約しているため日常利用への影響は軽微な見込みだが、`batch_runs` に実行記録を残し、レート制限エラーの頻度を監視する
- インフラ: GitHub Actions(無料枠 2,000分/月に対し、1日3回×数分で十分)、Neon 無料枠、Vercel Hobby(無料・非商用)→ **合計 $0/月**

### 7.2 成長時(モードB: Anthropic API)— 参考試算

移行判断の材料として残す。前提: 50記事/日、本文入力 ~4Kトークン/記事、出力 ~0.4Kトークン/記事。

| モデル構成 | 月額目安(通常) | Batch API(50%オフ)適用時 |
|---|---|---|
| Haiku 4.5($1/$5 per MTok) | ~$9 | ~$5 |
| Sonnet 5($3/$15) | ~$27 | ~$14 |
| Opus 4.8($5/$25) | ~$45 | ~$23 |

- プロンプトキャッシュ(固定システムプロンプト)でさらに1〜2割削減可能
- 移行時は `batch_runs` のトークン記録([05参照](./05_database-design.md))で実測して構成を決める

## 8. AI実行アダプタとAPI移行パス

digester / curator は LLM を直接呼ばず、共通インターフェース `LLMClient` 経由で呼び出す:

```
LLMClient.complete({ system, prompt, maxTokens }) → string(JSON文字列)
        │
        ├── ClaudeCodeAdapter(既定)
        │     `claude -p --output-format json` を子プロセス実行。
        │     認証: サブスクOAuth(ローカル)/ CLAUDE_CODE_OAUTH_TOKEN(CI。`claude setup-token` で発行)
        │
        └── AnthropicApiAdapter(成長時)
              `@anthropic-ai/sdk` Messages API。
              Structured Outputs / prompt caching / Batch API を有効化
```

- 切替は環境変数 `AI_PROVIDER=claude-code | api` のみ。呼び出し側(digester/curator)とプロンプト・zodスキーマは両モード共通
- 出力検証を zod に寄せているため、モードによる後段処理の差異はない
- **移行トリガー(いずれかを満たしたらモードBへ)**:
  1. 収益化する(サブスクの個人利用範囲を超えるため。規約面は [09参照](./09_legal-operational-concerns.md))
  2. 記事数・機能拡大で Pro のレート枠が普段の開発利用を圧迫し始めた
  3. Batch API / Structured Outputs 等のAPI専用機能が品質・運用上必要になった

## 9. 将来拡張との整合

| 将来機能 | 本設計での備え |
|---|---|
| タグ検索 | tags / article_tags を正規化して保存済み |
| 日別アーカイブ | daily_picks が日付キーで蓄積される |
| 既読・お気に入り(localStorage) | クライアント側のみで完結。API変更不要 |
| 重複グルーピング / 関連記事 | articles に埋め込みベクトル列を後付け可能(pgvector) |
| 週間まとめ | 蓄積済み要約を入力に新ジョブを追加するだけ |
| 通知 | daily_picks 確定をトリガーに配信ジョブを追加 |
